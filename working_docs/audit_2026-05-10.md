# LagBot — Deep Dive Audit

**Date:** 2026-05-10
**Scope:** Read of the repo at HEAD. No code changed. Covers how the repo works end-to-end, bugs in the opponent tracker, other env/wrapper bugs, missing tests, a critique of the self-play approach, and what a mature setup (Hydra/Pydantic configs, vec envs, eval suites, W&B, SkyPilot) would look like.

---

## 1. How the repo works today

**Engine** — `src/poker_env/` is a from-scratch Hold'em sim wrapped in `gymnasium.Env` (`TexasHoldemEnv`). Treys is only used for hand strength / showdown ranking, never for game flow. Game state machine lives in `game_state.py`; pot/side-pot math in `pot_manager.py`; player state in `player.py`. The big win is that `Player.starting_stack_this_hand` is snapshotted in `reset_for_new_hand()` before blinds, which is what makes the recent reward fix work.

**Observation** — 125 dims:
- 42 (hole + board cards w/ suit one-hot + present flag)
- 3 hand features (Treys MC equity, pot odds, SPR)
- 8 game-state
- **72 opponent features** = 9 slots × 8 features (`vpip, pfr, af/3, 3bet%, cbet%, fold_to_cbet%, wtsd%, confidence`). Zero-padded for missing seats.

This is the part you suspect is broken — and you're right.

**Action space** — `Discrete(6)`: fold / call / raise 0.5×pot / raise 1×pot / raise 2×pot / all-in.

**Training loop** — `train.py` builds `TexasHoldemEnv` + `OpponentAutoPlayWrapper` so SB3 only sees player 0's turns. The wrapper steps the env on the learning agent, then loops opponent agents (`OpponentPPO` / `CallAgent` / `RandomAgent`) until it's player 0's turn again or the hand ends. `create_opponents()` glob-finds the two most recent `final_model.zip` by `mtime` and uses them as opponents — generational self-play by directory timestamp.

**Backend/frontend** — FastAPI (`backend/`) wraps `TexasHoldemEnv` in `GameSession`. React (`frontend/`) talks REST + WS. Recently rewritten — not the source of training problems.

## 2. The opponent tracker is mostly broken

`src/poker_env/opponent_tracker.py::_update_opponent_stats` (lines 411-521) is the central function and it has structural bugs that make most stats wrong or meaningless.

### 2.1 `three_bet_opportunities` counts the wrong thing — line 456-462

```python
preflop_raises = [a for a in preflop_actions if a.action in [Action.RAISE, Action.ALL_IN]]
if preflop_raises:
    opponent.raised_preflop += 1
    opponent.three_bet_opportunities += 1   # <-- bug
    if len(preflop_raises) >= 2:
        opponent.three_bet_count += 1
```

A 3-bet opportunity is "someone else raised, and it's now your turn." This code only counts an opportunity when *the player themselves raised*. Players who fold facing a raise are invisible to the metric. The denominator is broken, so `three_bet_percent` is junk and feature index `[3]` going into the network is junk.

Also: `len(preflop_raises) >= 2` from a single player means they raised, got re-raised, then raised again — that's a 4-bet, not a 3-bet. A 3-bet should be detected from the hand's action sequence (was there a prior raiser when this player raised?), not from "did this one player raise twice."

### 2.2 WTSD is a heuristic that excludes losers — line 465-468

```python
if player_id in hand.winner_ids or (hand.actions and any(a.action == Action.FOLD for a in player_actions) and len(player_actions) >= 5):
    if len(flop_actions) > 0:
        opponent.went_to_showdown += 1
```

WTSD = "did this player get to showdown." This code increments **only if** they won the hand, or folded ≥5 actions in. A player who got to showdown and lost won't be counted. `wtsd_percent` is undercounted, `w$sd_percent` is overcounted (denominator = winners-who-saw-flop). The cleaner signal is `hand.betting_round == SHOWDOWN and player wasn't folded`.

### 2.3 Fold-to-c-bet — line 512-517

```python
if flop_actions and any(a.action in [Action.BET, Action.RAISE] for a in flop_actions):
    fold_actions = [a for a in flop_actions if a.action == Action.FOLD]
    if fold_actions:
        opponent.faced_flop_cbet += 1
        opponent.folded_to_flop_cbet += 1
```

This only fires when **the same player** both bet/raised the flop **and** folded the flop. That can't happen — once you fold you don't act again. So both counters always equal each other (or stay zero), and `fold_to_cbet_percent` is meaningless. Feature index `[5]` is dead.

The real metric needs hand-level context: did *someone else* c-bet, did this player face it, did they fold? You can't determine it from this player's actions alone.

### 2.4 C-bet opportunity has no preflop-aggressor check — line 504-509

```python
if len(flop_actions) > 0 and len(preflop_actions) > 0:
    if any(a.action in [Action.RAISE, Action.BET] for a in preflop_actions):
        opponent.flop_cbet_opportunities += 1
        if any(a.action in [Action.BET, Action.RAISE] for a in flop_actions):
            opponent.flop_cbet_made += 1
```

This counts any preflop raiser who saw a flop as having a c-bet opportunity. Correct, **but** also any flop bet they make counts as a c-bet, even if someone else bet first and they raised. A c-bet by definition is the preflop aggressor *opening* the flop betting.

### 2.5 Squeeze detection is wrong — line 495-501

```python
if preflop_actions and any(a.action == Action.RAISE for a in preflop_actions):
    action_types = [a.action for a in preflop_actions]
    if len([a for a in preflop_actions if a.action in [Action.CALL, Action.BET]]) > 0:
        opponent.squeeze_opportunities += 1
        if action_types[-1] == Action.RAISE:
            opponent.squeeze_attempts += 1
```

A squeeze is "raise after a raise *and* at least one caller, by a player who hadn't yet committed." This code is checking *this player's own* actions — if the player called then raised in the same preflop, both branches fire. Squeeze must be computed from the hand-wide action sequence at the moment the player acted.

### 2.6 Fold-to-3-bet-after-raising — line 482-491

```python
if opponent.raised_preflop > 0:
    if len(preflop_actions) >= 2:
        preflop_raises = [a for a in preflop_actions if a.action in [Action.RAISE, Action.ALL_IN]]
        if len(preflop_raises) == 1:
            opponent.faced_3bet_after_raise += 1
            if preflop_actions[-1].action == Action.FOLD:
                opponent.folded_to_3bet_after_raise += 1
```

`opponent.raised_preflop > 0` is a lifetime gate ("ever raised preflop in any hand"), so once it's true this branch will fire on every subsequent hand for that player. And the inner check uses only this player's actions, not whether somebody else 3-bet them. This stat is essentially noise after a few hands.

### 2.7 "Position" is `player_id`, not seat relative to button

`texas_holdem_env.py::_calculate_player_positions` line 262-267:

```python
for i, player in enumerate(self.game_state.players):
    positions[player.player_id] = i
```

The dict it records is `{player_id -> list_index}`, where the list index equals the player's seat number, not their offset from the dealer button. Since the button rotates and the players list doesn't get rotated, "position" never changes for a given player. Every position-based stat (the `position_stats` dict) is effectively keyed by `player_id`. "Position diversity" in the observation features is just "this opponent existed" — always 1/num_seats for everyone.

### 2.8 `Action.BET` is effectively never produced

`_string_to_action_enum` in the env will return `Action.RAISE` for any string containing "raise" before it reaches the "bet" branch — but more importantly, the action-space side never produces a string that triggers `Action.BET`. So every tracker branch keyed off `Action.BET` (and the `0.6` action-value encoding) is dead code. This means recent-actions encoding can never produce `0.6`, and the c-bet/flop-bet/squeeze detection treats all aggressive flop actions as `RAISE` instead of `BET`. Fine for now — but the code reads like both exist.

### 2.9 `_recalculate_stats` reads `recent_actions` deque (`maxlen=50`) for AF

Line 230-238: aggression factor is computed from the last 50 actions, but VPIP/PFR are lifetime. AF will swing wildly while VPIP/PFR don't, and the feature `af/3` (slot `[2]`) becomes a recency signal mixed with two cumulative signals. Decide on one regime (lifetime or windowed) and apply it consistently.

### Net effect on training

Of the 8 features per opponent fed into the network:
- Slots **3 (3-bet%), 4 (cbet%), 5 (fold-to-cbet%), 6 (wtsd%)** are wrong or biased
- Slot **2 (AF)** drifts with recency
- Slot **7 (confidence)** is just `min(hands/100, 1)`

The model is mostly learning from VPIP, PFR, and confidence — and even those are subtly affected by position being miscoded.

## 3. Other bugs and weirdness

### 3.1 Wrapper recomputes reward and discards intermediate shaping — `train.py:103-106, 129-133`

```python
if terminated or truncated:
    learning_agent = self.env.game_state.players[0]
    agent_starting_stack = learning_agent.starting_stack_this_hand
    reward = (learning_agent.stack - agent_starting_stack) / self.env.starting_stack
```

The env returns a reward that may include intermediate fold-shaping (±0.1 from the equity check). The wrapper *overwrites* it with stack delta whenever the hand ends. So on the step that ends the hand on the agent's own fold, the +0.1/−0.1 shaping is lost. If the hand ends during the opponent loop the wrapper also overwrites — but the env's `_execute_step` for that step returned reward for the opponent's action, which is correct to overwrite. The fix is to add the terminal stack delta to the shaping reward (`reward = reward + (stack - start)/starting_stack` only when the wrapper is replacing an opponent-caused reward).

### 3.2 `num_players` config drift

`configs/default_config.yaml` says `num_players: 6`. `train.py` line 332 hard-codes `num_players=3` when building the env. Every "default" config is silently overridden. Heads-up + 1 opponent only — not 6-max as advertised.

### 3.3 Opponent-profit attribution is split equally — `train.py:164`

```python
per_opponent_profit = total_profit / len(self.opponents)
```

When you win a hand against two opponents, only one of them actually called you down. Splitting equally smears the signal. Without per-pot contestant info, the dashboard "opponent profit" view is misleading.

### 3.4 Generational self-play by `mtime` is fragile

`create_opponents` (in `train.py`) globs `models/**/final_model.zip` and grabs the two most recent by mtime. Touching a file, rsync'ing, or re-saving a checkpoint shuffles the opponents silently. There's no record of which two opponents a run was trained against — runs aren't reproducible.

### 3.5 `TrainingCallback.save_freq` uses `self.n_calls`, not `num_timesteps`

With multi-env training (you'll want this eventually), `n_calls` counts callback invocations, not env steps. Currently you have a single env so it's equivalent, but flip to `SubprocVecEnv` and the save cadence quietly changes.

### 3.6 Dead duplicate code

`TexasHoldemEnv.step_with_raise()` (the original raise-sizing path) is still in the file but unreferenced — easy to confuse a future reader.

## 4. Where tests are missing

Tests today cover `test_reward_bugs.py` (great), basic VPIP/PFR/AF (`test_opponent_tracker.py`), wrapper smoke (`test_wrapper_rewards.py`), and side-pot scenarios (`test_all_in_comprehensive.py`). The gaps that matter:

| Priority | Test | Why |
|---|---|---|
| P0 | 3-bet detection across hand action sequence | `three_bet_*` is wrong (§2.1) |
| P0 | Fold-to-c-bet across players (A bets, B folds) | Current logic is impossible to trigger (§2.3) |

| P0 | WTSD includes losers who saw showdown | Undercounts (§2.2) |

I'd write the P0 tests *first*, *before* touching the tracker — they'll define the contract you fix to.

## 5. Critique of the self-play approach

Your current approach: train vs `CallAgent + RandomAgent`, then iteratively swap in the latest PPO checkpoint as the opponent, expecting steady improvement. There are three problems:

**a) Only two opponents at a time, picked by mtime.** Your model never plays a *distribution* of past selves — it plays the two newest. That's not curriculum, it's "always exploit the latest". You can train a model that beats the very latest checkpoint while losing to one from five generations ago (rock-paper-scissors cycles are common in self-play).

**b) No held-out evaluation.** There's no fixed eval suite ("vs CallAgent for 1000 hands, vs RandomAgent for 1000 hands, vs the gen-1 PPO snapshot for 1000 hands") run on a schedule. Your only signal is `mean_reward` against the *training* opponents — which is the same signal the model is gradient-descending into, so it's circular.

**c) Bad opponent features means PPO can't actually exploit.** Even if the meta-loop were perfect, ¾ of the opponent features are wrong (§2). The model has no reliable read on opponent tendencies. So "self-play that exploits past opponents" can't happen mechanically.

### What more mature setups do

- **Opponent pool sampling** (PFSP / fictitious self-play): keep a pool of all past checkpoints, sample one per episode with weights ∝ `1 - win_rate_vs_them` (so you mostly play the ones you're losing to). DeepMind's *AlphaStar*, Pluribus, and most poker bots use a variant.
- **Tournament eval**: every K checkpoints, round-robin tournament across a fixed eval set (gen1, gen5, gen10, CallAgent, RandomAgent, ε-Nash-ish baseline). Plot `BB/100` vs each opponent. That's your real progress signal.
- **Reproducible opponent selection**: record opponent paths + hashes in the run's metadata. Don't rely on `mtime`.

For this codebase the smallest first step is:
1. Fix the tracker bugs so feature inputs are correct
2. Write a `tournament.py` that takes a list of model paths and plays a round-robin
3. Make `train.py` accept an explicit `--opponents` list (with a sampling strategy flag) instead of glob-by-mtime

## 6. What a mature setup looks like, and SkyPilot

### Configuration

You have `configs/*.yaml` loaded with `yaml.safe_load`. That works until two configs drift apart (which has already happened — see §3.2). Two common upgrades:

- **Pydantic models** for config. Define a `TrainingConfig`, `EnvConfig`, `PPOConfig`. Reject unknown keys. Catches typos at startup, not at minute 10 of training.
- **Hydra** if you want CLI overrides like `python train.py training.lr=3e-4 env.num_players=6` and config composition (`+opponent=pool_sample`, `+opponent=fixed_v2`). Heavier dependency but very common in RL.

The litmus test: can someone re-run your last training run from a single commit hash + config file, and get the *same opponents*? Today no — fix that.

### Vec envs

`SubprocVecEnv([make_env() for _ in range(N)])` with `VecMonitor`. PPO scales near-linearly with parallel envs up to your CPU count. With `n_steps=3000` per env × 8 envs you get a rollout of 24k transitions per update; gradient quality goes up a lot, and wall time per timestep goes down. Be aware: opponent agents inside the env need to be pickle-safe (loading a frozen SB3 model in each subproc is fine).

### Eval

SB3's `EvalCallback` runs a separate env on a schedule and logs `eval/mean_reward`. Couple this with a custom callback that runs your tournament every N steps and logs `tournament/bb_per_100/vs_<opponent>` per opponent. That's the dashboard you actually want.

### Experiment tracking

TensorBoard is fine for scalar curves. For comparing runs, configs, and artifacts: **Weights & Biases** or **MLflow**. Both work with SB3 (`from wandb.integration.sb3 import WandbCallback`). At minimum log: config snapshot, opponent paths + hashes, every eval result, final model.

### Model registry

Stop trusting `mtime`. Move models into a structure like `models/<run_name>/<step>.zip` with a `manifest.json` that has run config hash, parent checkpoint, training timesteps, eval scores. `create_opponents` should read manifests, not glob files.

### SkyPilot for cloud GPU

SkyPilot is a great fit because:
1. PPO + a Python game env is single-node — no orchestration nightmare
2. You can run on spot for 70-80% off
3. It abstracts AWS/GCP/Azure/Lambda

A minimal `sky.yaml` for this repo would look like:

```yaml
resources:
  accelerators: A10:1            # or A100, L4 — PPO is CPU-bound a lot of the time
  use_spot: true
  cpus: 16+
  memory: 32+

file_mounts:
  /workspace: .                   # rsync the repo
  /workspace/models:
    name: lagbot-models           # cloud bucket for checkpoints
    mode: MOUNT

setup: |
  cd /workspace
  pip install -r requirements.txt

run: |
  cd /workspace
  python train.py \
    --config configs/default_config.yaml \
    --name $SKYPILOT_TASK_ID
```

Then `sky launch -c lagbot sky.yaml`. Checkpoints land in the bucket, so spot preemption + auto-recovery (`sky launch --retry-until-up`) is safe. For multi-seed sweeps: `sky jobs launch` with a `params` matrix.

Two prerequisites before you do this:

1. **GPU usage actually matters.** Your env is pure Python and runs on CPU. PPO's gradient step is GPU-accelerated, but with `n_steps=3000 × 1 env` you barely benefit from a GPU. Move to `SubprocVecEnv` first — then cloud GPU pays for itself.
2. **Bucket-backed model dir.** Today the code writes to `./models`. On SkyPilot you want that under a mounted bucket so it survives preemption and so a follow-up run can read prior opponents.

## 7. Recommended order of work

1. **Lock down the opponent-tracker contract with P0 tests** (3-bet, c-bet, fold-to-cbet, WTSD). Write them against the *desired* behavior. They'll fail. Good.
2. **Fix `_update_opponent_stats`** by reasoning about hand-level action sequences, not per-player action lists. Make the tests pass.
3. **Fix `_calculate_player_positions`** to record seat-relative-to-button.
4. **Fix wrapper reward overwrite** to preserve intermediate shaping (§3.1).
5. **Fix `num_players` config drift** — read it from YAML.
6. **Replace `create_opponents` mtime glob** with an explicit `--opponents` list / manifest registry.
7. **Add a `tournament.py`** + eval callback; plot real progress curves.
8. **Move to `SubprocVecEnv`**, then to SkyPilot.

Items 1-4 will probably matter more for your win rate than any other change. The model isn't learning a bad strategy — it's learning from corrupted opponent features and a noisy reward signal.
